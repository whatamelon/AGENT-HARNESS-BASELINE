# UI 데모 작성 가이드

`05-ui-demo.html` / `05-ui-demo.md` 를 작성할 때 따르는 규칙. Step 5 진행 중 이 문서를 참조한다.

## 1. 레이아웃 원칙

- **상하 스택**: As-Is 위, To-Be 아래 — 좌우 비교보다 각 화면을 크게 볼 수 있다.
- **사이드 메모**: 각 화면(mock) 오른쪽 260px 폭에 상태/변경 메모 컬럼.
- **화면 블록 반복**: 기능이 여러 화면에 영향을 주면 `section.screen-block` 을 복제.

## 2. 하이라이트 규칙

To-Be 쪽 요소에만 변경 종류를 색으로 표시한다. As-Is는 원형 그대로 — 비교 기준이 되어야 한다.

| 변경 종류 | 클래스 | 스타일 | 예시 |
|---|---|---|---|
| 추가 | `.added` | 녹색 dashed border | 신규 버튼, 신규 필드 |
| 변경 | `.changed` | 노란 solid border | 레이아웃 재배치, 텍스트 변경 |
| 제거 | `.removed` | 빨간 strikethrough | 삭제된 버튼·필드 |

```html
<button class="added">신규 저장 버튼</button>
<span class="changed">변경된 라벨</span>
<span class="removed">기존 중복 필드</span>
```

한 요소에 두 가지가 겹칠 때는 **가장 의미 있는 하나만** 선택. 지나치게 많은 하이라이트는 노이즈.

## 3. 컨텍스트 소스 매핑

| 섹션 | 출처 문서 | 추출 기준 |
|---|---|---|
| 핵심 변경 요약 | `04-plan.md` §1 "최종 결정 사항" | 3~5개, 사용자에게 체감되는 것 우선 |
| As-Is 화면 | `01-analysis.md` §2 "관련 코드 맵", §3 "아키텍처 현황" | 현재 파일에서 UI 요소 역재구성 |
| As-Is 메모 | `01-analysis.md` §6 "제약·리스크", §7 "분석 소견" | 문제점·불편함 기술 |
| To-Be 화면 | `02-design.md` §3.1 "사용자 흐름", §3.2 "컴포넌트 설계" | 설계된 UI 요소 재구성 |
| To-Be 메모 | `04-plan.md` §1, §2 | 의도·기대 효과 기술 |

## 4. 화면 모킹 스타일

- **정적 HTML만** — JS/React/Vue 금지. md-to-pdf의 Chrome 렌더는 JS 실행을 보장하지 않는다.
- **Tailwind/외부 CSS 금지** — 인라인 스타일 또는 템플릿의 기본 스타일만.
- **실제 컴포넌트 충실도가 아닌 **구조적 재현**에 집중** — 테이블 헤더/필드명/버튼 위치가 맞으면 충분. 픽셀 퍼펙트 금지.
- **텍스트는 실제 복사본 사용** — 영문 대체("Button") 대신 실제 레이블("주문 확정").

## 5. As-Is 없는 신규 기능 처리

`01-analysis.md`에 "현재 미구현" 으로 기록된 경우 As-Is 블록을 아래 중 하나로 대체:

**패턴 A — "해당 기능 없음"**:
```html
<div class="screen-mock" style="display:flex;align-items:center;justify-content:center;color:#9ca3af;">
  해당 기능 없음 — 신규 도입
</div>
```

**패턴 B — "기존 우회 방법"**: 사용자가 현재 어떻게 우회하고 있는지를 표시. 엑셀 입력, 수기 기록, 타 시스템 등.

As-Is 메모는 반드시 "현재 사용자가 겪는 불편" 을 명시해야 To-Be의 가치가 드러난다.

## 6. 05-ui-demo.md 구조

HTML만 덩그러니 있으면 PDF 목차에서 검색이 안 되므로 마크다운 래퍼로 감싼다:

```markdown
# UI 변경 데모: <기능명>

> 생성일: YYYY-MM-DD
> 기반: 01-analysis.md, 02-design.md, 04-plan.md

## 핵심 변경 요약

<!-- 04-plan.md §1에서 3~5개 -->

- 변경 1
- 변경 2
- 변경 3

## 시각 비교

<!-- 05-ui-demo.html의 <body> 내부를 그대로 임베딩 (중복되는 <h1 class="demo-title"> 블록은 제거) -->

## 화면별 상세

### 화면 1: <이름>

**As-Is**: <문장형 설명>
**To-Be**: <문장형 설명>

- <변경 1>
- <변경 2>
```

## 7. 좋은/나쁜 예

| ❌ 나쁨 | ✅ 좋음 |
|---|---|
| 모든 요소에 `.changed` 하이라이트 → 무엇이 핵심인지 불명 | 실제 변경된 2~4개에만 하이라이트 |
| Tailwind `class="bg-blue-500 p-4"` 사용 | 인라인 스타일 or 템플릿 클래스만 |
| 사이드 메모 "UI가 개선됩니다" | "주문 확정 버튼을 헤더로 이동 → 스크롤 없이 즉시 접근" |
| As-Is 블록에 하이라이트 적용 | As-Is는 중립, To-Be에만 하이라이트 |
| 한 HTML에 10개 화면 쌓기 | 핵심 3~5개 화면만 — 나머지는 메모로 대체 |
