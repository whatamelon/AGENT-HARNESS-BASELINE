사용자가 에러 수정을 요청했습니다: **$ARGUMENTS**

---

## 개요

에러를 받으면 **바로 수정하지 않는다.** 먼저 트라이지로 접근 방식을 결정하고, 충분히 파악한 뒤 계획을 사용자에게 보여준 후 구현한다.

```
[0] 컨텍스트 확인  →  [T] 트라이지  →  [1] 에러 재현·수집  →  [2] 딥다이브 RCA  →  [3] 수정 계획  ⏸  →  [4] 구현·검증  →  [5] 결과 보고서
                                                                                                          ↑
                                                                                               사용자 승인 후 진행
```

**기본 동작**: `--gate` 미지정 시 Step 3(수정 계획) 완성 후 멈추고 사용자 승인을 요청한다. 구현은 승인 후에만 시작.

## 외부 스킬 적용

- **DB 디버깅**: supabase-postgres-best-practices
- **재현/회귀 테스트**: vitest
- **Next.js 이슈**: next-best-practices

## 가벼운 디버깅이 필요하면

간단한 버그 분석은 `/debug`를 사용한다 (원인 분석 + 수정 제안만). `/feature-error-fix`는 **트라이지·RCA·보고서 등 공식 산출물이 필요한 복잡한 버그**에 사용한다.

## 사전 준비 — 조건부 Reference Read

> 버그 유형에 따라 필요한 reference 파일만 Read한다. 이미 현재 세션에서 Read한 파일은 재읽기 불필요.

| 버그 유형 | 필수 Read |
|-----------|-----------|
| 모든 버그 | `.claude/references/architecture.md` |
| DB/데이터 관련 | + `.claude/references/db-schema.md` |
| API/서버 에러 | + `.claude/references/api-design.md` |
| 프론트엔드 에러 | + `.claude/references/frontend-guide.md` |
| 비즈니스 로직 | + `.claude/references/business-rules.md` |

## Step 0. 인자 파싱 및 컨텍스트 확인

### 0-1. 인자 파싱

`$ARGUMENTS`에서 다음을 추출한다:

| 인자          | 기본값 | 설명                                                            |
| ------------- | ------ | --------------------------------------------------------------- |
| `<에러 설명>` | (필수) | 에러 증상, 메시지, 또는 `.feature/` 폴더 경로                   |
| `--gate`      | `plan` | 어디서 멈출지: `rca`(분석만), `plan`(계획까지), `fix`(구현까지) |
| `--source`    | (선택) | 에러 로그, 스택 트레이스, 관련 파일 경로 등 직접 전달할 소스    |

### 0-2. 기존 산출물 확인

`.feature/` 폴더에서 관련 산출물을 찾는다:

1. `$ARGUMENTS`가 `.feature/` 경로를 직접 포함하면 해당 폴더 사용
2. 아니면 `.feature/` 하위에서 키워드로 매칭되는 `fix-` 폴더 탐색
3. 매칭되는 폴더가 있으면 기존 산출물을 읽어 현재 단계를 파악하고 이어서 진행
4. 없으면 새로 생성

### 0-3. 출력 폴더 결정

```
.feature/YYYY-MM-DD-fix-<kebab-case-에러명>/
├── 00-triage.md         ← Step T 산출물
├── 01-error-report.md   ← Step 1 산출물
├── 02-rca.md            ← Step 2 산출물
├── 03-fix-plan.md       ← Step 3 산출물
└── 04-fix-report.md     ← Step 5 산출물
```

- 날짜는 오늘 날짜 사용
- 폴더명 prefix는 반드시 `fix-`로 시작
- 같은 날짜+이름이 이미 존재하면 `-2`, `-3` 접미사 추가

## Step T. 트라이지

**목적**: 에러를 받자마자 바로 분석하거나 수정하지 않는다. 먼저 상황을 파악해 어떤 경로로 접근할지 결정한다. 이 판단이 이후 모든 단계의 깊이와 방향을 결정한다.

### T-1. 초기 정보 평가

`$ARGUMENTS`와 `--source`에서 파악 가능한 내용을 정리한다. 부족한 정보가 있으면 **지금 사용자에게 질문**한다 — 정보가 부족한 상태에서 진행하면 잘못된 방향으로 깊이 파고들 수 있다.

확인할 항목:

- **에러 메시지 / 스택 트레이스**가 있는가? → 있으면 발생 위치 파악 가능
- **재현 가능**한가? → "항상 발생" / "간헐적" / "특정 조건"
- **언제부터** 발생했는가? → 최근 배포/커밋과 연관 가능성
- **영향 범위**가 얼마나 큰가? → 단일 사용자 / 전체 / 특정 기능

정보가 불충분하면 사용자에게 다음을 요청한다:

```
아래 정보를 알려주시면 더 정확하게 분석할 수 있습니다:
1. 에러 메시지 / 콘솔 로그 (있다면)
2. 언제부터 발생했나요? (특정 작업 후? 항상?)
3. 재현 절차가 있나요?
```

### T-2. 에러 유형 분류

수집한 정보로 에러 유형을 판단한다:

| 유형                 | 특징                                                    | 접근 방식                              |
| -------------------- | ------------------------------------------------------- | -------------------------------------- |
| **A. 명확한 버그**   | 스택 트레이스 있음, 항상 재현, 원인 코드 위치 추정 가능 | 빠른 트랙: RCA → 바로 수정 계획        |
| **B. 불명확한 버그** | 간헐적 발생, 원인 레이어 불명확, 여러 가설 경쟁         | 정밀 트랙: 재현 조건 확정 → 체계적 RCA |
| **C. 회귀 버그**     | 이전에 동작했던 기능이 특정 시점 이후 깨짐              | 커밋 이분 탐색 → 변경 diff 분석        |
| **D. 긴급 장애**     | 프로덕션 영향, 즉각 조치 필요                           | Workaround 먼저 → RCA는 이후           |

### T-3. 긴급도 판단

- **긴급 (D유형)**: Workaround가 가능하면 먼저 제시하고 사용자 확인 후 RCA 진행
- **일반**: Step 1부터 순서대로 진행

### T-4. 트라이지 요약 출력

`00-triage.md`에 작성하고 사용자에게 간략히 보고한다:

```markdown
# 트라이지: <에러명>

- Treat this skill as a shared Claude Code ↔ Codex workflow. If source text mentions `.claude/`, Claude-only slash commands, or Claude-only tools, map them to the nearest project docs, `AGENTS.md`, `.codex/`, `.omc/`, native CLI, or available Codex skill/plugin surface.
- Prefer native CLI first for GitHub/Git work (`gh`, `git`) and do not publish issues, PRs, commits, Slack messages, or external side effects without explicit user intent/approval in the current task.
- Flag machine-specific absolute paths, usernames, local archives, and environment-specific assumptions as portability risks.
- Verify with concrete commands or generated evidence before claiming completion.


> 작성일: YYYY-MM-DD

## 에러 유형

- 분류: A(명확) / B(불명확) / C(회귀) / D(긴급)
- 재현 가능 여부: ...
- 영향 범위: ...

## 접근 방식

<!-- 어떤 경로로 진행할지, 이유 포함 -->

## 필요한 추가 정보

<!-- 없으면 "충분함" -->

<!-- 초기 판단: UI / Server Action / Service / DB / 외부 API -->
```

트라이지 후 사용자에게 1~2줄로 보고한다:

> "B유형(불명확한 버그)입니다. 간헐적 발생이라 재현 조건부터 좁히겠습니다."

## Step 1. 에러 재현 및 수집

이 단계의 목적은 에러를 **정확히 이해하고 재현 가능한 상태로 만드는 것**이다. 원인 분석은 아직 하지 않는다 — 지금은 사실(fact)만 수집한다.

### 1-1. 에러 정보 수집

아래 항목을 최대한 수집한다. 없는 항목은 "확인 불가"로 표기한다:

**증상**

- 에러 메시지 전문 (스택 트레이스 포함)
- 발생 위치 (파일, 함수, 라인 번호)
- 발생 환경 (브라우저, OS, Node 버전, 브랜치 등)
- 발생 빈도 (항상 / 간헐적 / 특정 조건에서만)

**재현 조건**

- 재현 절차 (step-by-step)
- 재현에 필요한 데이터나 상태
- 재현이 안 되는 경우 — 마지막으로 정상 동작했던 시점 또는 커밋 확인:
  ```bash
  git log --oneline -10
  ```

**영향 범위**

- 영향받는 기능 목록
- 영향받는 사용자 또는 데이터 범위
- 회피 방법(workaround)이 존재하는지

**관련 코드 위치**

- 에러가 발생하는 파일/컴포넌트
- 에러 메시지 기준으로 관련 파일 검색
- 최근 변경 이력 확인 (에러 발생 전후 커밋):
  ```bash
  git log --oneline --since="3 days ago" -- <관련파일>
  ```

### 1-2. 에러 재현 보고서 작성

수집한 내용을 `01-error-report.md`에 작성한다:

```markdown

## 1. 에러 증상

| ...  | ...   | ...           |

<!-- 에러 발생 전후 주요 커밋 -->

<!-- 이 시점에서 직관적으로 의심되는 원인. RCA에서 검증할 대상 -->
```

## Step 2. 딥다이브 근본 원인 분석 (RCA)

에러 보고서를 기반으로 코드를 탐색하여 **진짜 원인**을 찾는다. 증상이 아니라 원인을 수정해야 하므로, 이 단계를 건너뛰면 같은 에러가 다른 방식으로 재발한다.

### 2-1. 가설 목록 작성

`01-error-report.md`의 정보를 바탕으로 가능한 원인 가설을 나열한다:

- 가설은 구체적이어야 한다 ("상태 관리 문제" X → "X 컴포넌트에서 Y 상태가 unmount 후에도 업데이트됨" O)
- 가설마다 확인 방법을 함께 정의한다
- 가능성 높은 순서로 정렬한다

**가설이 여러 개이고 방향이 불명확하면** 사용자에게 보여주고 확인을 받는다:

> "현재 가설이 2개입니다: (A) ... (B) ... A부터 검증할게요, 괜찮으신가요?"

### 2-2. 레이어 경계 체크

이 프로젝트는 FSD 7-레이어 + 모놀리식 서버 로직(Server Actions + Route Handlers) 구조를 사용한다. 에러가 어느 레이어에서 발생했는지, 어느 레이어에서 잘못된 데이터/상태가 생성됐는지 경계를 하나씩 확인한다.

**레이어별 확인 방법:**

| 레이어                                | 확인 포인트                                   | 도구                             |
| ------------------------------------- | --------------------------------------------- | -------------------------------- |
| **UI (Pages / Widgets / Components)** | 컴포넌트에 전달된 props/state 값, 렌더링 조건 | React DevTools, console.log      |
| **Server Actions / Route Handlers**   | 입력값 파싱, 리턴값 구조, 에러 처리 여부      | Next.js 서버 로그, 타입 확인     |
| **비즈니스 로직 (Actions 내부)**      | 도메인 규칙 적용, 데이터 변환 로직            | 코드 읽기, 단위 테스트           |
| **데이터 (Supabase / 외부 API)**      | 쿼리 결과, RLS 정책, 컬럼 타입 불일치         | Supabase 대시보드 로그, MCP 쿼리 |

**경계 확인 순서**: 에러 스택 트레이스에서 시작해 **위쪽(원인)으로 거슬러 올라간다.** 에러 메시지가 UI에서 났더라도, 잘못된 값이 만들어진 곳은 Server Action이나 데이터 레이어일 수 있다.

### 2-3. 가설 검증 기법

각 가설을 확인할 때 아래 기법을 상황에 맞게 선택한다:

**코드 정적 분석** (기본):

- 관련 파일의 로직 흐름 추적
- 타입/인터페이스 불일치 확인 (레이어 간 계약 위반)
- 최근 변경 diff 확인:
  ```bash
  git show <커밋해시> -- <관련파일>
  git diff HEAD~5 -- <관련파일>
  ```

**런타임 추적** (정적 분석으로 원인을 못 찾을 때):

- 의심 지점에 console.log 추가 — 값이 실제로 어떤 상태인지 확인
- 로그 추가 후 사용자에게 재현을 요청하거나 직접 재현 가능하면 실행
- 로그 삽입 위치: 에러 발생 지점 바로 위 → 한 레이어 위 → 진입점 순으로 거슬러 올라감

**이분 탐색** (C유형 회귀 버그):

- 정상 동작하던 커밋과 현재 사이에서 원인 커밋을 이진 탐색:
  ```bash
  git log --oneline <정상커밋>..<현재브랜치> -- <관련파일>
  git show <중간커밋> -- <관련파일>
  ```

**데이터 추적** (Data 레이어 의심 시):

- Supabase MCP로 실제 쿼리 결과 확인 (`/db-query` 스킬 활용)
- RLS 정책이 데이터를 필터링하고 있는지 확인
- 컬럼 타입과 코드의 TypeScript 타입이 일치하는지 확인

**막혔을 때**: 한 가설을 20분 이상 검증했는데 확인도 기각도 안 되면 멈추고 사용자에게 알린다. 잘못된 방향으로 계속 파는 것보다 방향을 바꾸는 게 낫다.

### 2-4. 근본 원인 확정

모든 가설 검증 후 근본 원인을 하나로 좁힌다. 여러 원인이 복합된 경우 각각을 명시한다.

`02-rca.md`에 작성한다:

```markdown
# 근본 원인 분석 (RCA): <에러명>

> 작성일: YYYY-MM-DD
> 기반: 01-error-report.md

## 1. 가설 검증 결과

| #   | 가설 | 검증 방법 | 결과              | 비고 |
| --- | ---- | --------- | ----------------- | ---- |
| 1   | ...  | ...       | ✅ 확인 / ❌ 기각 | ...  |

## 2. 근본 원인

- **위치**: `파일경로:줄번호`
- **원인**: ...
- **왜 이 코드가 에러를 유발하는가**: ...
```

## Step 3. 수정 계획

근본 원인이 확정되면, **어떻게 수정할지** 계획을 세운다.

### 3-1. 수정 옵션 도출

가능한 수정 방법을 2~3가지 도출한다:

- **즉각 수정 (Quick Fix)**: 최소 변경으로 에러를 막는 방법. 기술 부채가 남을 수 있음.
- **근본 수정 (Proper Fix)**: 원인을 제거하는 올바른 방법. 변경 범위가 클 수 있음.
- **우회 수정 (Workaround)**: 원인은 두고 증상만 처리. 임시방편으로만 고려.

각 옵션마다 평가:

- 변경 범위 (파일 수, 레이어 수)
- 기존 동작에 대한 영향
- 기술 부채 발생 여부
- 테스트 용이성

### 3-2. 구현 계획 수립

`03-fix-plan.md`에 작성한다:

```markdown
# 수정 계획: <에러명>

> 작성일: YYYY-MM-DD
> 기반: 02-rca.md

## 1. 수정 옵션 비교

| 옵션      | 설명 | 변경 범위 | 기존 동작 영향 | 기술 부채 | 선택 |
| --------- | ---- | --------- | -------------- | --------- | ---- |
| 즉각 수정 | ...  | ...       | ...            | ...       | ❌   |
| 근본 수정 | ...  | ...       | ...            | ...       | ✅   |

## 2. 선택 근거

## 3. 구현 계획

### 변경 파일 목록

| 파일 | 변경 유형      | 변경 내용 요약 |
| ---- | -------------- | -------------- |
| ...  | 수정/신규/삭제 | ...            |

### 구현 순서

1. ...

### 회귀 방지 전략

- 확인할 기능 목록: ...
- 실행할 테스트: ...

## 4. 재발 방지 조치

- [ ] ...
```

### 3-3. ⏸ 사용자 승인 요청 (`--gate plan` 기본값)

계획서를 완성했으면 **반드시 멈추고** 사용자에게 보고한다:

```
📋 수정 계획이 준비됐습니다: .feature/YYYY-MM-DD-fix-<에러명>/03-fix-plan.md

근본 원인: <한 줄 요약>
수정 방법: <선택한 옵션 한 줄>
변경 파일: <N>개

구현을 시작할까요?
```

`--gate fix`로 명시된 경우에만 이 단계를 건너뛰고 바로 Step 4로 진행한다.

## Step 4. 구현 및 검증

사용자가 승인하면 수정 계획에 따라 실제로 코드를 수정하고 검증한다.

### 4-1. 구현

`03-fix-plan.md`의 구현 계획을 따른다. 프로젝트 기본 규칙 준수:

- FSD 레이어 + 모놀리식 서버 로직 준수
- 파일명 kebab-case, 컴포넌트 PascalCase
- 절대 경로 import (`@/...`)

**수정 범위 원칙**: 에러 수정에 필요한 변경만 수행한다. 주변 코드 정리나 리팩토링은 별도 작업으로 분리한다.

### 4-2. 빌드 검증

```bash
npm run build
npm run lint:fix
```

빌드 실패 시 수정 후 재검증.

### 4-3. 에러 재현 확인

`01-error-report.md`의 재현 절차를 다시 실행하여:

- 에러가 더 이상 발생하지 않는지 확인
- 수정 전과 동일한 조건에서 정상 동작하는지 확인

### 4-4. 회귀 확인

`03-fix-plan.md`의 "회귀 방지 전략"에 명시한 기능들을 확인한다:

- 단위 테스트가 있는 경우: `npm test`
- UI 확인이 필요한 경우: `/e2e-qa-report` 스킬 활용 가능

## 참고: 이터레이션

이미 `.feature/YYYY-MM-DD-fix-<에러명>/` 폴더가 존재하고 사용자가 같은 에러에 대해 재호출하면:
