## §1 요구사항 조견표

두 단계 테이블로 구성한다.

### 마스터 테이블

원본 요청서 또는 `04-plan.md` §1 "최종 결정 사항"에서 요구사항을 추출한다:

```markdown
## 1. 요구사항 — 구현 조견표

| # | 요구사항 | PR | 구현 상태 | 검증 결과 | 비고 |
|---|---------|:---:|:--------:|:--------:|------|
| [A] | {{요구사항 설명}} | #N | ✅ 완료 / ⏸️ 보류 / ❌ 미구현 | ✅ 통과 / — | {{비고}} |
```

구현 상태 기호:
- ✅ 완료 — 코드 작성 + 빌드 통과
- ⏸️ 보류 — 의도적 제외 (사유 비고에 기재)
- ❌ 미구현 — 누락 또는 실패

### 세부 항목 테이블

마스터 테이블의 각 요구사항을 세부 항목으로 분해한다:

```markdown
### 요구 N — {{요구사항 제목}}

| 세부 항목 | 구현 | 검증 |
|----------|:----:|:----:|
| {{세부 기능 1}} | ✅ | ✅ 스크린샷 확인 |
| {{세부 기능 2}} | ✅ | ✅ |
```

"스크린샷 확인"은 §2에서 해당 캡처가 존재할 때 표기한다.

## §2 E2E 스크린샷

화면별로 캡처 이미지를 임베딩하고 확인 포인트를 기술한다:

```markdown
## 2. E2E 검증 스크린샷

### 화면 N: {{화면 이름}}

![{{화면 설명}}](_screenshots/NN-kebab-name.png)

- {{확인 포인트 1}}
- {{확인 포인트 2}}
```

규칙:
- 이미지 경로는 **상대 경로** (`_screenshots/...`)
- 데스크탑은 fullPage 캡처, 모바일은 viewport 캡처
- 각 캡처마다 2~4개 확인 포인트 (무엇이 정상 동작하는지)

## 작성 원칙

- **조견표가 본질**: 요구사항 대비 구현/검증 상태를 빠짐없이 추적. 누락된 요구사항이 있으면 ❌로 명시.
- **스크린샷은 증거**: 조견표에서 "스크린샷 확인"으로 표기한 항목은 반드시 §2에 해당 캡처가 존재해야 한다.
- **이미지 임베딩은 상대 경로**: md-to-pdf가 같은 디렉토리 기준으로 해석한다. 절대 경로 금지.
- **기존 에러 구분**: 빌드/린트에서 기존 이슈와 우리 코드 이슈를 명확히 분리.
